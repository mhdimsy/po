# قوانین عملیاتی Codex

## 1. محدوده پروژه

این پروژه یک Digital Twin Prototype برای تولید کارخانه است، نه سیستم عملیاتی واقعی کارخانه.

نسخه اول:

- بدون AI/ML واقعی
- بدون Docker
- بدون Login
- Local Dev
- SQL Server لوکال
- Backend: FastAPI + SQLModel
- Frontend: React + Vite + Tailwind
- ورودی کارخانه: CSV جداگانه
- داده‌های انسانی، مواد، supplier، سناریو و event داخل سیستم ساخته می‌شوند

## 2. قانون محدوده تسک

هر تسک فقط همان چیزی را پیاده کند که در فایل تسک آمده است.
اگر چیزی در تسک نیست، پیاده‌سازی نکن.

مثلاً اگر تسک import است، simulation ننویس.
اگر تسک backend است، UI ننویس.
اگر تسک validation است، import واقعی ننویس.

## 3. قبل از کدنویسی

برای هر تسک اول این‌ها را بده:

- برداشت از تسک
- فایل‌هایی که قرار است ایجاد/ویرایش شوند
- مدل یا API پیشنهادی اگر مربوط است
- ریسک‌ها یا ابهام‌ها

بعد از تأیید کاربر کدنویسی کن.

## 4. بعد از کدنویسی

بعد از هر تسک باید این فایل‌ها را در صورت نیاز آپدیت کنی:

```text
logs/IMPLEMENTATION_LOG.md
logs/CHANGELOG.md
logs/DB_SCHEMA_CHANGE_LOG.md
logs/TESTING_LOG.md
logs/KNOWN_ISSUES.md
logs/OPEN_QUESTIONS.md
docs/PROJECT_STATUS.md
```

همچنین اگر ماژول جدید ساختی، برایش یک سند در `docs/modules/` بساز.

## 5. قانون مستندسازی تغییرات

هر تغییر باید توضیح داشته باشد:

- چرا انجام شد؟
- کجا انجام شد؟
- چطور اجرا/تست می‌شود؟
- چه چیزی عمداً انجام نشده؟

## 6. محدودیت‌های مهم

- AI/ML در V1 پیاده‌سازی نشود.
- Message Broker پیاده‌سازی نشود.
- Docker اضافه نشود.
- Login/Auth اضافه نشود.
- Cost Model در V1 خاموش است.
- داده واقعی کارخانه مستقیم خوانده نمی‌شود؛ CSV import داریم.
- CSVها شبیه نام ستون‌های دیتابیس فعلی کارخانه هستند.
- warning در validation هم blocker است.

## 7. اگر ابهام داشتی

اگر ابهام جزئی است، بهترین تصمیم محافظه‌کارانه را بگیر و در `logs/OPEN_QUESTIONS.md` ثبت کن.
اگر ابهام روی معماری یا دیتابیس اثر جدی دارد، قبل از کدنویسی از کاربر بپرس.

# سیاست مستندسازی پروژه

Codex بعد از هر تسک باید مستندات را آپدیت کند. مستندات بخشی از خروجی تسک هستند، نه کار اضافه.

## فایل‌هایی که همیشه بررسی می‌شوند

### 1. `logs/IMPLEMENTATION_LOG.md`

برای ثبت اجرای هر تسک:

- Task ID
- تاریخ
- خلاصه کار
- فایل‌های تغییرکرده
- تصمیم‌های فنی
- تست‌های انجام‌شده
- کارهای باقی‌مانده

### 2. `logs/CHANGELOG.md`

برای تغییرات قابل مشاهده پروژه.

### 3. `logs/DB_SCHEMA_CHANGE_LOG.md`

اگر جدول، ستون، migration یا مدل دیتابیس تغییر کرد، این فایل باید آپدیت شود.

### 4. `logs/TESTING_LOG.md`

هر تست دستی/خودکار و نتیجه آن اینجا ثبت شود.

### 5. `logs/KNOWN_ISSUES.md`

خطاها، نقص‌ها و محدودیت‌های شناخته‌شده.

### 6. `logs/OPEN_QUESTIONS.md`

ابهام‌هایی که بعداً باید از کاربر پرسیده شود.

### 7. `docs/PROJECT_STATUS.md`

وضعیت کلی پروژه، تسک‌های انجام‌شده، تسک بعدی.

## مستندات ماژولی

برای هر ماژول جدید، یک فایل در این مسیر بساز:

```text
docs/modules/<module-name>.md
```

مثلاً:

```text
docs/modules/import_factory.md
docs/modules/master_data.md
docs/modules/simulation.md
```

هر سند ماژول باید شامل این‌ها باشد:

- هدف ماژول
- موجودیت‌ها
- APIها
- سرویس‌ها
- وابستگی‌ها
- محدودیت‌های فعلی
- تست‌های پیشنهادی

## ADR

اگر تصمیم معماری مهم گرفتی، ADR جدید بساز:

```text
adrs/ADR-XXX-title.md
```

از قالب `templates/ADR_TEMPLATE.md` استفاده کن.
